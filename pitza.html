Pitza

헌혈이 필요한 사용자와 헌혈 가능한 사용자를 연결하는 매칭 플랫폼입니다. 실시간 채팅과 위치 기반 조회 기능을 중심으로 사용자 간 빠른 연결과 상호작용을 구현한 프로젝트입니다.

2025.03 – 2025.06 팀 프로젝트 백엔드 중심 참여

프로젝트 개요

  • 헌혈이 필요한 사용자와 참여 가능한 사용자를 연결하는 매칭 서비스
  • 실시간 상호작용을 중심으로 사용자 간 빠른 의사소통이 가능하도록 설계
  • 위치 기반 조회를 통해 주변 헌혈원 정보를 직관적으로 탐색할 수 있도록 구현

화면 구성

주요 기능

  • 매칭 기능 · 사용자 선호 정보를 기반으로 헌혈 요청자와 참여 가능 사용자를 연결
  • 실시간 채팅 · Socket.IO 기반 메시지 송수신 처리 및 즉시성 있는 사용자 간 상호작용 구현
  • 위치 기반 조회 · Kakao Map API를 활용한 주변 헌혈원 탐색 기능 제공
  • 파일 관리 · MinIO를 활용한 이미지 및 데이터 저장 구조 구성
  • 실시간 데이터 처리 · Redis를 활용한 채팅 메시지 처리 및 상태 관리

트러블슈팅

1. WebSocket 환경에서 세션 기반 사용자 인증 동기화 문제

문제
  • WebSocket 연결 시 브라우저 로그인 세션이 함께 전달되지 않아 서버에서 사용자 식별 불가
  • 별도의 session_key를 클라이언트에서 전달받는 방식에 의존 → 조작 가능성 존재
원인
  • WebSocket은 HTTP와 다른 방식으로 동작하여 기존 세션 인증 흐름이 자동으로 적용되지 않음
  • CORS 환경에서 credential 설정이 맞지 않으면 쿠키가 전달되지 않음
해결
  • 클라이언트/서버 credential 설정을 통일하여 쿠키 전달 보장
  • Socket 인증 미들웨어에서 handshake header의 cookie 추출
  • Django 사용자 조회 API를 통해 세션 검증
  • 검증된 사용자 정보를 socket.data.user에 저장하여 모든 이벤트에서 재사용
결과
  • 별도 userId/session_key 없이 서버 기준으로 사용자 식별 가능
  • 실제 로그인 세션과 일관된 실시간 인증 환경 구축
  • 클라이언트 조작 가능성 제거 및 인증 신뢰도 향상

2. 실시간 메시지 중복 처리 제거 및 브로드캐스트 구조 재설계

문제
  • 메시지 전송 시 발신자에게도 동일한 데이터가 전달되어 클라이언트에서 중복 처리 발생
  • 발신/수신 구분 로직이 클라이언트에 분산되어 UI 복잡도 증가
  • 텍스트/이미지 이벤트 구조가 분리되어 메시지 처리 흐름이 일관되지 않음
원인
  • io.to(roomId).emit() 사용으로 발신자를 포함한 전체 사용자에게 메시지 전송
  • 메시지 유형별로 이벤트 이름과 처리 로직이 분산된 구조
해결
  • socket.broadcast.to(roomId).emit()으로 변경하여 발신자를 제외한 사용자에게만 전송
  • 메시지 이벤트를 단일 구조로 통합하여 text/image 처리 흐름 일관화
결과
  • 클라이언트 중복 처리 및 분기 로직 제거
  • 서버 중심으로 메시지 흐름을 통제하여 유지보수성 향상
  • 이벤트 구조 표준화로 기능 확장 용이

회고

Pitza 프로젝트를 통해 실시간 통신, 위치 기반 조회, 파일 저장 기능을 하나의 서비스로 통합하며 기능 간 흐름을 설계하는 경험을 했습니다. 특히 실시간 채팅 구현 과정에서, 백엔드는 단순히 데이터를 처리하는 역할을 넘어서 사용자 경험과 서비스의 동작 방식을 결정하는 중요한 계층이라는 점을 체감했습니다.
