트러블슈팅

1. WebSocket(STOMP) 환경에서 JWT 인증 정보 유지 및 사용자 식별 문제

문제
실시간 토론 기능 구현 중 WebSocket(STOMP) 연결 내에서 JWT 인증 정보가 연동되지 않아 `headerAccessor.getUser()`가 `null`이 되는 문제가 발생했습니다. 이로 인해 사용자 식별 및 권한 검증이 불가능했습니다.
원인
WebSocket은 HTTP Handshake 이후 별도의 프로토콜로 동작하여 Spring Security의 JWT 필터 체인이 적용되지 않습니다. 또한 SockJS 환경에서는 CONNECT 이후 메시지 프레임에서도 인증 정보가 자동으로 유지되지 않았습니다.
해결
`ChannelInterceptor` 기반 StompAuthChannelInterceptor를 도입했습니다.
1) Handshake 단계: URL 파라미터의 토큰을 세션에 저장
2) CONNECT 단계: Header 또는 세션에서 토큰을 추출·검증 후 Principal 설정
3) 이후 프레임: 세션의 인증 정보를 복원하여 사용자 컨텍스트 유지
결과
WebSocket 환경에서도 일관된 사용자 인증 체계를 구축했습니다.
이를 통해 토론방 입장, 상태 변경, 메시지 전송 전 과정에서 권한 기반 로직을 안정적으로 적용할 수 있었습니다.

2. ElastiCache TLS 설정 불일치로 인한 Redis Timeout 및 서비스 중단 문제

문제
배포 후 Django 기반 AI 기능이 정상 동작하지 않으며 서비스가 중단되었습니다. Redis 연결 시 타임아웃이 발생했지만, 즉시 예외가 발생하지 않고 로그가 남지 않아 원인 파악이 어려운 상황이었습니다.
원인
AWS ElastiCache의 전송 중 암호화(TLS) 옵션이 활성화되어 있었으나, 애플리케이션에서는 평문 Redis 연결을 사용하고 있었습니다. 이로 인해 TCP 연결은 정상적으로 이루어졌지만, 실제 Redis 요청에서는 응답을 받지 못해 timeout이 발생했습니다.
해결
ECS 컨테이너 내부에서 직접 테스트를 수행했습니다.
1) TCP 연결은 정상임을 확인하여 네트워크 문제를 배제했습니다.
2) Redis ping 테스트에서만 timeout이 발생하는 점을 통해 TLS 미적용 문제를 원인으로 특정했습니다.
3) Redis 연결을 rediss:// 기반으로 변경하고, Redis.from_url()을 사용해 환경별 설정을 통합했습니다.
결과
Redis 연결 문제를 해결하여 서비스 중단 현상을 복구했습니다. 또한 타임아웃 설정을 명시적으로 추가해 예외가 즉시 기록되도록 개선하고, 로컬/운영 환경을 모두 지원하는 구조로 정리해 재발 방지 기반을 마련했습니다.
